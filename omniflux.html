OmniFlux - Single-Page Wiki

You're looking at a complete, self-contained wiki. You can download the file and edit it using the built-in Markdown editor; try it now by pressing the "Edit" button in the upper right. Learn about supported Markdown.

Below are different versions of this same wiki. You can preview them by following the links. If you'd like to download them, right-click on the link and "Save As".

  • The full site (what you are viewing now): unminified or minified. This is currently about 32k unminified, 20k minified due to the content.
  • A blank wiki, without extra content: unminified or minified. The blank file is only 20k unminified and about 8k minified.
  • Or you can build your own version by downloading the GitHub repository.

In general, I would highly suggest the unminified version. However, if you really need a few extra kilobytes of space, the minified one is just fine.

Features

  • Lighter (though it's also less feature-packed) than TiddlyWiki and Feather Wiki.
  • Markdown support (italics, bold, both, code, strike), external images, links, ordered and unordered lists, code fences. Learn more!
  • Generated HTML isn't one long line, allowing tools like git to work better.
  • Automated tests to ensure the bidirectional HTML to/from Markdown conversion works as expected.
  • Doesn't need JavaScript to browse, thanks to CSS-based navigation and sidebar, allowing it to work even in a very restrictive environment.
  • Quick editing and saving using Ctrl+Enter or Cmd+Enter.

Markdown Syntax

Markdown is supported, though not all features are included. This page will explain all of your available options, including common patterns that are not supported.

When something is marked as "not supported", you could contribute code through a pull request.

Basic formatting

Use italics, bold, and both. This works evenwithinwords.

Paragraphs are one long line. These will become a single paragraph. Do not use newlines inside of a paragraph unless you want the paragraph broken onto another line, like the following message:

When you use newlines
Then a break will be added
Make your own haiku

Avoid

Using _underscores_ for emphasis (bold, italics, both) will not work.

Headings

Headings 1 through 6 are supported. Put the corresponding number of # symbols at the beginning of a line.

Avoid

The alternate method of using a line of ------ or ====== underneath the heading is not supported.

Lists

  1. Ordered lists are supported.
  2. Just prefix each list item with a number and a period.
  • Unordered lists are also supported.
  • Use a hyphen for each item

Avoid

Nested lists do not work. Nesting other markup, such as paragraphs or blockquotes is also not supported.

Starting the numbering of a list with any number other than 1 is not supported.

Using + or * for list items will work, but they will be converted to -.

Task lists, which use [] and [X] for a list of empty or filled checkboxes is not supported.

Images and Links

URLs are converted automatically: https://fidian.github.io/omniflux

Link to an external page: OmniFlux

Link to an internal page: Main Page or Markdown Info

External image: (the "alt text" is also converted to a title)

Linking around an external image:

Avoid

Links can have titles by adding a description after the URL, but that's unsupported.

Do not use angle brackets (< and >) around a URL to turn it into a link. Similarly, URLs without http:// or https:// in front are not linked.

Email addresses are not linked automatically.

Reference-style links are not supported, which is where a link is used in one or more spots, but the URL is maintained elsewhere.

Code

You can inline a block of code with backticks.

Code fences are supported and language information is preserved.

While code fences support language information, no highlighting happens at this stage in order to keep the wiki small. If you want it, you will need to include Prism or highlight.js yourself.

Avoid

Code fences can specify a language, but no special highlighting happens.

Indenting a block of code with 4 spaces will not turn it into a code block.

Backticks within a code block are not supported.

Completely Unsupported

Embedding HTML within the Markdown. This is a tricky thing to do, especially since the conversion needs to accurately be bidirectional.

Single blockquotes and nested blockquotes will not work.

Horizontal rules via ---.

Underline via wrapping with ~, delete via wrapping with ~~~.

Anchors, like #[jump-here] since the wiki uses CSS-based navigation that relies on the location hash and there's a concern about duplicating a page ID and a user-generated ID.

Tables, along with aligned cells within the table.

Footnotes.

Rule Processing

When converting Markdown to HTML, there are a bunch of rules that need to get applied in a particular order. Also, sometimes rules need to build on themselves, such as when an image is linked. The rule processing engine is a generic way to process a string through a set of rules and have handlers decide how each interaction unfolds.

The Markdown first will look for block-level markup. Each time a chunk is found, that chunk is processed fully. This means the HTML is created to handle the block level element and any inline elements within the block.

Inline elements are a bit more tricky due to possibly wanting recursion and sometimes not. For instance, using backticks for code means anything within needs to be excluded from further processing, but one could have a bolded word in a link.

Rule Definitions

Each rule is governed by a pattern and a handler. When the pattern matches text, the match result is passed in as arguments to the handler. This supports groups. Take, for example, this rule for strikeout text, or the second rule for erasing newlines.

[/~~(.+?)~~/, (_, txt) => ["<s>", txt, "</s>"]]

[/\n+/, () => ""]

If the pattern was matched against the string "one ~~two~~ three", then the handler's _ variable will be "~~two~~" and the txt variable will be "two".

Each handler needs to return one of two things:

  • An odd-length array of strings. The first, third, etc. index in the array would be strings that are "final" and that should not be processed further. All of the other indices are for strings that could still be modified by additional handlers. This is what the strike-through rule does.
  • A string. When this happens, the matched pattern is exchanged for the string. The second rule that eliminates newlines does this, so if passed in "abc\nxyz", then that string would be modified to be "abcxyz".

Markdown Processing Quick Overview

First, block handlers will look for block-level elements. When a match is found, it must handle the block and all of the inlines within. Typically a block handler will match the beginning and end of a section of a document and then will pass chunks of text to an inline processing function.

The inline one is a bit harder because some things are expected to have further processing and some are not. For instance, inline code blocks (with backticks) are not supposed to get any further processing within the backticks. Images do get further processing because they might be turned into a link. Knowing if the handler should return an array or a string is vital.
